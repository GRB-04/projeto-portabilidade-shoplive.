4. Principais Achados
⚠️
Layout com largura fixa de 1250px (Versão A)
O container principal da seção de live usa width: 1250px com o player em width: 900px e o chat em width: 300px. Em qualquer tela menor que 1250px, o conteúdo transborda criando scroll horizontal — tornando o botão de compra inacessível no mobile, o que representa perda direta de receita em um cenário real de Black Friday.
⚠️
Chat Widget com right: -20px (Versão A)
O widget de chat foi posicionado com right: -20px, o que intencionalmente deixa 20px do widget fora da tela. Em mobile, isso corta o botão de fechar e o campo de input, impedindo o usuário de interagir com o chat — a principal ferramenta de engajamento em live commerce.
✅
Player de vídeo com aspect-video (Versão B)
A Versão B usa a classe Tailwind aspect-video (equivalente a aspect-ratio: 16/9) combinada com w-full. Isso garante que o player sempre mantém a proporção correta independente da largura disponível, sem hardcode de pixels — a técnica correta para vídeos responsivos.
✅
Docker garante ambiente idêntico em qualquer SO
Ambas as versões rodam via imagem multi-stage Docker (node:20-alpine para build + nginx:alpine para serve). Isso elimina a síndrome do "na minha máquina funciona" — o comportamento de runtime é byte-a-byte idêntico em Windows, macOS e Linux.
5. Conclusão

✅ Versão B aprovada para produção

A Versão B demonstrou portabilidade completa em todos os dispositivos testados — de 320px (mobile entry-level) a 2560px (4K). As correções aplicadas seguem as melhores práticas da indústria: layouts fluidos com flexbox e CSS Grid, proporções de mídia com aspect-ratio, e componentes adaptáveis via breakpoints do Tailwind CSS. A Versão A, como esperado, falhou em todos os cenários mobile, comprovando que código sem planejamento responsivo é inviável para plataformas de live commerce com audiências mistas.
